# 二次说话人切分校准日志说明

## 背景：为什么需要二次校准

### 行业难题：短声纹识别

短声纹识别（<2s）确定说话人身份一直是行业难题。声学特征在极短时间内提取的嵌入向量不够稳健，传统 speaker verification 方案在面对几百毫秒的短语音时几乎无能为力。

但在视频对话翻译领域，我们不应该继续把说话人判别当成传统的 speaker verification 问题。传统方案只看声学特征，而对话场景中存在大量额外的判别信号：**连续性**（同一人倾向于连续说话）、**上下文**（对话有来有回）、**语义**（语气词回应与前方提问的关联）、**声学**（声纹相似度）。四者联合判别，才能在短语音场景下做出可靠决策。

基于这个思路，我们构建了一条非常灵敏的 diarization 管线，能切分出对话中短于 300ms 的中间人喊话。

### ffmpeg 伪影问题

我们发现用 ffmpeg 将声音从视频中分离出来后，不同的码率（kbps）会在人耳无法分辨的高频/低频区域留下伪影。二次精细 diarization 模型非常敏感，在一些类型的视频场景中，这些伪影会引发说话人误判。

各种尝试后，发现最有效的方案是：**带通滤波**去除伪影频段 → **声纹 + LLM 语义**自动校准残余的误判。

### CB 和 CP 是如何产生的

首先，我们使用了一个端到端模型，能够同时通过声学特征 + 语义特征一次性输出 ASR 结果和说话人标签分割。这个模型的优势是**语义块**检测比较准确，能够避免 VAD 把语音切得过于细碎的问题。

但在快节奏对话（300-500ms 间隔，如吵架、辩论、庭审等场景）下，端到端模型仍然会搞错说话人。错误主要表现为两种模式：

#### CB（跨边界对）：语义块边界的说话人归属错误

端到端 ASR 模型以语义块为单位输出结果。当快节奏对话中出现短促回应紧贴另一说话人长句时，模型容易把回应错误归入前方说话人的语义块——因为它"跨越了语义块边界"。

以 `示例项目/逐玉` 为例，端到端模型原始输出如下：

```json
{
  "start_ms": 0,
  "end_ms": 2760,
  "text": "你说的，你又何必着急着认呢？欸？",
  "speaker_id": 0
},
{
  "start_ms": 2760,
  "end_ms": 11150,
  "text": "你个臭丫头。哎呦，外人怎么想我不介意，可我对你这么多年的情谊，难道只能用那些黄白之物做衡量吗？对。",
  "speaker_id": 1
},
{
  "start_ms": 11150,
  "end_ms": 12720,
  "text": "就只能这么衡量。",
  "speaker_id": 0
}
```

注意第一个语义块末尾的 **"欸？"** 和第二个语义块末尾的 **"对"**——这两个短促回应实际上都是第二说话人（speaker_id: 1）说的，紧贴前方说话人的长句，端到端模型把它们错误地包含进了前方说话人的语义块中。

CB 校准就是检测这类跨语义块边界的说话人归属错误，把错误归入的短句划回正确的说话人。

#### CP（切分点对）：语义块内部的误切

经过 diarization 模型精细化二次切分后，很多短插话或音色接近的片段基本都能被正确分割出来。但精细切分也可能导致误切——把同一说话人的连续表达切成了两段。我们称这种需要校准的相邻子句对为**切分点对（CP）**。

> CP 部分的音频示例后续补充。

## 日志文件

使用 `--asr-mode precise`（高级模式默认值）时，系统会在 ASR 阶段执行二次说话人切分，并通过 LLM + 声纹对 CB 和 CP 进行自动校准。校准完成后，自动在 `segments/ASR/` 下生成日志文件：

```
示例项目/逐玉/segments/ASR/
└── secondary-diarization-calibration-log.md    ← 二次说话人切分校准日志
```

## 日志内容

日志包含三部分：

### 1. 概览统计

```
- 原始子句数: 11
- 合并后子句数: 9
- 减少句数: 2
- 跨边界对(CB): 4 个评估, 2 个合并
- 切分点对(CP): 8 个评估, 0 个合并
```

- **CB（跨边界对）**：跨语义块边界的相邻子句，判断短句是否被错误归入前方说话人
- **CP（切分点对）**：同一语义块内部经 diarization 二次切分后的相邻子句，判断切分是否为误切

### 2. 逐句评估详情

每个子句会标注评估结论和依据：

```
- 子句2 [2.513~2.757 (0.24s)]: 哎
  ↓ 合并 (来源: cross_boundary_llm) — 同一人连续表达，先感叹后斥责，语义连贯
```

结论分两种：

| 结论 | 含义 |
|------|------|
| **保留** | 保持切分，两个子句属于不同说话人或语义断开 |
| **合并** | 合并为一句，两个子句是同一说话人的连续表达 |

依据来源：

| 来源 | 说明 |
|------|------|
| `cross_boundary_llm` | LLM 判定跨边界子句应合并 |
| `cross_boundary_rule` | 规则判定跨边界子句不应合并（如间隔超时） |
| `cut_point_llm` | LLM 判定切分点合理（应保留） |
| `cut_point_rule` | 规则判定切分点（如声纹相似度极低=不同说话人） |

### 3. 合并前后对比

逐句展示合并结果，方便对照检查：

```
[1] 2.513~4.118 — 合并

合并前:
- 哎
- 你个臭丫头。

合并后: 哎你个臭丫头。
```

## 什么时候会触发

仅 `--asr-mode precise` 模式下才会生成此日志。`--asr-mode basic` 模式使用 ASR 自带的说话人切分，不经过二次评估。
